## Integration testing and system audits

### Integration tests for critical paths

We built an integration testing layer. These are not unit tests with mocks. Each test spins up a real database context, creates actual records through the service layer, and verifies the outcome against the database state.

The first batch covers five areas:

**Post creation.** The test creates a user, then creates a post through the tRPC `post.createPost` procedure. It verifies the post exists in the database with the correct `userId`, `visibility` set to `PUBLIC`, topic association, and that the content stored matches what was sent. It also verifies that the post appears in the user's `posts` relation.

**Feed queries.** Two tests here. The home feed test creates multiple posts across different users and topics, then calls the feed procedure with each filter mode (latest, best, follows) and verifies the returned posts match the expected set and order. The profile feed test verifies that querying a specific user's profile returns only their posts, correctly paginated.

**Post interactions.** The test creates two users, has user A create a post, then has user B like it and reply to it. It then verifies three things: the `UserReactionsOnPosts` record exists with the correct `reaction` enum value, the reply post has its `parentPostId` pointing to the original, and a notification record was created for user A with `type` set to `REPLY` and the correct `sourceUserId`.

**User profiles.** Creates a user, populates their profile fields, follows another user, creates posts. Then fetches the profile through the tRPC procedure and verifies follower/following counts, post count, and that all profile fields round-trip correctly.

**Moderation.** Creates a post, triggers a moderation report, then simulates the voting flow with multiple users voting to hide. Verifies the vote records, the threshold calculation, and the resulting visibility change on the post.

These tests run against a test database that gets reset between runs. They exist so we can refactor subscription and referral logic with confidence that we haven't broken the social layer underneath.

### Trials and referrals audit

We traced the full trial subscription flow and referral reward chain across the codebase. The goal was to map every dependency between these two systems before modifying either one.

The trial flow starts in `subscription.service.ts` where `startTrial` creates a user record with `plan: TRIAL_PERIOD` and sets an expiry date. The referral flow starts in the signup procedure, where a `referralCode` from localStorage gets attached to the new user. When a referred user upgrades their plan, `referral.service.ts` calls `upgradeReferrerToFreePlan` to reward the referrer.

The problem: these two flows are coupled at seven points. The referral reward function checks `plan === TRIAL_PERIOD` before granting rewards, meaning if trials are disabled (which they currently are), no referral rewards can ever fire. The `invite-friends` page has a CTA button whose click handler is an empty function, literally `() => {}`. The `referralRewardsBalance` field on the User model is still being read in profile queries but nothing ever writes to it. The `REFERRAL_REWARD_CREDITS` constant is set to 0. The function `upgradeReferrerToFreePlan` contains dead conditional branches that reference plan tiers that no longer exist in the enum.

We documented all seven issues and filed them. The core finding is that referrals need to work independently of trial status. Right now they can't.

### Reputation system audit

We audited the profile reputation system by reading every file involved: `reputation-rules.service.ts`, `reputation.service.ts`, `credit-distribution.service.ts`, and the `calculate-weekly-reputation` Inngest job.

Three of five pillars have real implementations. Quality calculates a score from reactions received on the user's posts using a weighted map (Smart and Useful count +3, Inspiring +1, Deceptive and Aggressive count -5) and normalizes the raw sum through `Math.tanh(rawScore / 30) * 100` to keep it in the -100 to 100 range. Wellbeing counts five distinct activity types during the epoch (reactions given, moderation votes, original posts, comments, poll votes) and scores 100/80/50/0 based on how many types the user engaged in. Altruism does the same count-based scoring but for tips given.

Respect and Competence are stubs. Both functions exist but return 0. Since the global score is the average of all five pillars, two zero-value pillars permanently drag every user's score down by 40%.

We also found a discrepancy in reaction weights. The design spec defines AGREE as 0 points (neutral acknowledgment) but the code has it at +1. DISAGREE is -1 in both. This makes positive reactions slightly overweighted relative to the original design intent.
