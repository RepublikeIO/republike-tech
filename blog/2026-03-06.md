## Fixing what users see, testing what they don't

### Shipped

#### Image display bug resolved

Some posts on the feed were showing broken or missing images. Users would see empty spaces where images should be, or link previews stuck on a placeholder instead of showing the actual article thumbnail.

Two separate issues were causing this.

**Link preview images always showing the placeholder.** When a post contains a link to an external article, we display a preview card with the article's title and image. The code was checking the dimensions of the preview image to decide whether to show it or fall back to a placeholder. The problem: it was reading the image dimensions immediately after setting the `src`, before the browser had actually downloaded the image. At that point, `width` and `height` are both zero, so every image failed the size check and got replaced by the placeholder. We rewrote this to use proper `onload`/`onerror` callbacks. The dimension check now runs only after the image has fully loaded, so valid preview images display correctly.

**Post images disappearing on initial page load.** Our image optimization function rewrites image URLs to go through Next.js's image optimizer, which handles resizing and format conversion. This function needs to know the current domain to process same-origin images. It was reading `window.location.hostname` directly, but on the server during the initial page render, `window` doesn't exist. This caused a crash during server-side rendering, which produced a mismatch between what the server sent and what the browser expected. React detected the mismatch and re-rendered the page from scratch, causing a visible flash where images would briefly disappear. We added a server-side guard so the function skips domain-based URL rewriting during the initial render and only applies it once the page is running in the browser.

Both fixes are live. Posts now display their images correctly on first load, and link previews show the actual article thumbnail.

#### Automated visual testing with Playwright

We set up Playwright, a browser automation framework, to run automated tests against our production-like environment. These tests open a real browser, navigate the app, and verify that what users see matches what we expect.

Four test suites are now in place:

- **Homepage smoke test.** Loads the public landing page and verifies that the hero section, pricing table, footer, and navigation all render without errors.
- **Hydration consistency test.** Loads authenticated pages and listens for rendering mismatches between the server and the browser. This is the automated safety net for the image display bug we just fixed. If a similar issue ever resurfaces, this test catches it before it reaches users.
- **Link preview test.** Loads the feed, finds posts with link previews, and verifies the preview image is the real article thumbnail rather than our placeholder. It also checks that the image actually loaded by verifying its pixel dimensions are greater than zero.
- **Post image integrity test.** Scans all visible images in the feed and verifies none are broken. Every image must have loaded successfully and be going through our optimization pipeline.

These tests run against a local instance with production data. They can be launched headless for fast automated runs or with a visual UI for debugging. This gives us the ability to catch visual regressions across different screen layouts that are difficult to spot through manual testing alone.

#### Trial period reactivation via Stripe

We completed our investigation into how to reactivate the free trial period for users who want to try a membership for a week. The trial system was disabled several months ago and is coupled with the referral reward system at multiple points. We have now mapped the full flow through Stripe's subscription API, identified how to offer a seven-day trial without conflicting with existing subscription states, and documented the edge cases around what happens when a trial expires or when a user who already had a trial wants to restart. Implementation begins next week.

#### Mobile CI pipeline

We re-implemented the continuous integration pipeline for the mobile application. The CI automatically builds, tests, and validates each code change before it can be merged. Having this pipeline in place shortens the feedback loop significantly. Developers know within minutes whether their changes break anything, instead of finding out days later during manual testing. This is especially important for the mobile app where each release goes through app store review. Catching issues early means fewer rejected builds and faster iteration on new features.
