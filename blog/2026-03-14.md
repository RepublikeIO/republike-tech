# Friday, March 14th 2026

## Preparing the trial period

### Investigation and planning across three codebases

This week was about laying the groundwork to reactivate the free trial for new subscribers. The trial system was disabled months ago. Turning it back on isn't a switch flip — it touches the backend, the web frontend, and the mobile apps, and each has its own set of gaps.

We started by auditing every trial-related file in the webapp backend. The database schema already has all the fields: `trialEndDate`, `trialSource`, `trialCancelAtEnd`, `trialAutoRenew`. The trial service has full lifecycle management — start, cancel, convert, expire. Apple and Google webhook handlers know how to detect trial purchases and route them correctly. What's missing is the web path through Stripe.

The payment router explicitly rejects trial subscriptions. Line 53: `throw new Error('Trial period subscription has been removed')`. The `createSubscription()` function in the Stripe service never passes `trial_period_days` to the Stripe API. There's a `startTrial()` function in the subscription service that sets all the right database fields, but it's not exposed through any tRPC endpoint. The frontend filters the trial plan out of the paywall display.

So the infrastructure is there but every entry point is bricked.

For the web, we're going with Stripe-native trials. The user provides a payment method, gets seven days free, and Stripe auto-charges when the trial ends. This is different from the mobile approach where Apple and Google handle billing through their own intro offer systems. Stripe managing the billing means we don't need a cron job to expire web trials — Stripe fires `invoice.payment_succeeded` when it charges, and `customer.subscription.deleted` if the user cancels. The existing cron job will continue handling mobile-only trials.

We created two epics for the webapp. The backend epic has seven issues: adding `trial_period_days` support to Stripe subscription creation, exposing a `startTrial` tRPC endpoint, updating the payment router guard, handling the `customer.subscription.trial_will_end` webhook for reminder emails three days before charge, updating the expiration cron to skip Stripe-managed trials, aligning mobile product ID detection, and verifying the invoice webhook correctly transitions trial users to paid subscribers.

The frontend epic has four issues: unhiding the trial plan in the paywall, building a checkout flow that collects a payment method without an immediate charge, displaying trial status and remaining days in the app, and handling the expiration transition with contextual messaging instead of a generic paywall redirect.

On the mobile side, we filed thirteen issues across iOS and Android. The main ones: removing dead `startTrial` mutation code that's never actually called (the button bypasses it and goes straight to IAP), fixing the Android offer ID filter that blocks the trial plan from displaying, passing `appAccountToken` on iOS and `obfuscatedAccountId` on Android so webhooks can reliably match purchases to users, adding a trial countdown banner, and implementing receipt refresh for status synchronization.

Each issue has acceptance criteria, file paths, line numbers, and dependency chains. The backend dependency order is clear: Stripe subscription changes first, then the endpoint, then webhooks and cron updates. Frontend work starts once the endpoint exists. Mobile work is mostly independent.

### Community bug: reply threading depth limit

A community member reported they couldn't reply to comments beyond a certain depth. We investigated and found the nesting limit is hardcoded to three levels across four component files. The reply button disappears at level three with no explanation. One of the handler functions is literally `console.warn('Not yet implemented')`.

Whether this is a deliberate design choice or an unfinished feature is ambiguous — the code explicitly disables the button at depth three, but the placeholder handler and lack of any user-facing message suggest it was meant to be extended. We filed the issue with three options ranging from a quick fix (add a "reply limit reached" message and a "continue in new thread" button) to a full refactor with recursive components and virtualized threads. This one is in progress.
