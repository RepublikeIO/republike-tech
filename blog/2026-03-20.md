## Simulating the future, reconnecting the pipes

### Shipped

#### Algorithmic differentiation threshold simulator

We built an interactive tool in the admin dashboard that answers a specific question: does Republike have enough data for different content algorithms to produce meaningfully different results?

The context is that we're exploring the possibility of offering multiple feed algorithms — user-proposed ranking rules, a governance-selected algorithm, or options from an external marketplace. But algorithms need data to work with. If the network produces 20 posts a day, every algorithm will show the same 20 posts in roughly the same order. The algorithm becomes cosmetic. We needed a way to quantify where we stand relative to the point where algorithmic choice actually matters.

The simulator uses six metrics: daily active users, posts per day, interactions per day, content diversity, session frequency per user, and social graph density. Each metric has a minimum threshold (below which the metric contributes nothing useful to an algorithm) and an ideal value (where the metric fully supports algorithmic differentiation). The thresholds come from recommendation systems literature — collaborative filtering requires around 20 interactions per user to produce actionable recommendations, A/B tests need roughly 1,600 samples per variant to detect a 5% effect with 80% statistical power, and community detection algorithms need an average node degree above 10 to identify meaningful clusters.

We queried the production database to seed the simulator with real data. The current state of the network: 11 daily active users, 24 posts per day, 18 interactions per day, 4.5 average connections per user. The resulting readiness score is around 4%. The simulator makes it immediately visible that we are far below the threshold where introducing multiple algorithms would produce a user-perceptible difference.

Five preset profiles let you compare stages of growth — from current state through early traction, algorithmic readiness, sustained growth, and mature network. An info button opens a modal explaining the theoretical basis for each metric, with references and an honest disclaimer that the exact numbers are estimates, not universal constants.

The tool includes a language toggle between French and English.

#### Admin dashboard CI/CD restored

The admin dashboard had no automated deployment. The last Vercel deployment was 220 days old, done manually. The Vercel project existed but was not connected to the GitHub repository.

We linked the repository to the existing Vercel project. Pushes to `main` now trigger automatic production deployments, and pull request branches generate preview deployments. This matches the setup already in place for the main webapp. The admin dashboard is now continuously deployed at its existing Vercel URL.

### In Progress

#### Trial period reactivation — web and mobile

We're implementing the free trial flow across the web application and the mobile apps. On the web side, this involves the Stripe subscription API — creating a seven-day trial subscription without conflicting with existing subscription states, handling trial expiration, and managing the edge case of users who previously had a trial. The investigation completed two weeks ago mapped all the coupling points between trials and the referral system, and we're now building the actual implementation.

On the mobile side, the trial flow needs to work through the App Store and Google Play in-app purchase systems, which have their own trial period mechanics and receipt validation requirements. The goal is a consistent trial experience regardless of platform.
